## 基础类型

基础数据类型，或者叫原始数据类型。在 JavaScript 中， 共有8种内置数据类型：包括7种基本数据类型和1种引用数据类型。

- 基本数据类型：`Null`、`Undefined`、`Boolean`、`Number`、`String`、`BigInt`、`Symbol`。

- 引用数据类型：`Object`。

TypeScript 是 JavsScript 的超集，因此也提供了与原始类型对应的数据类型：`null`、`undefined`、`boolean`、`number`、`string`、`bigint`、`symbol`、`Object`。

::: tip
JavaScript 的数据类型是大写，而 TypeScript 对应的类型是小写，只有 `Object` 是大写。
:::

### boolean、number、string、bigint

```typescript
const b1: boolean = true;
const b2: boolean = false;

const n1: number = 3; // 整数数字
const n2: number = 0.001; // 小数数字
const n3: number = 0o77; // 八进制数字
const n4: number = 0x1f; // 十六进制数字
const n5: number = 0b00101; // 二进制数字

const bi1: bigint = BigInt(100); // 通过 BigInt 函数创建 bigint 类型变量
const bi2: bigint = 100n; // 通过字面量语法创建 bigint 类型变量

const s1: string = "hello world";
```

### null 和 undefined

`null` 和 `undefined` 的使用包含两种情况。是否开启 `--strictNullChecks` 编译选项。

- 启用 `strictNullChecks` 时，`undefined` 和 `null` 只能赋值给各自类型的变量。

  ```typescript
  const u: undefined = undefined;
  const n: null = null;
  ```

- 不启用 `strictNullChecks` 时。可以赋值给除 `never` 类型外的其他任何类型的变量。（不推荐）

  ```javascript
    // 可以赋值给除尾端类型外的任何类型。当然，还包括自定义类型，这里没有列出来
  let a1: boolean = undefined;
  let a2: number = undefined
  let a3: string = undefined
  let a4: undefined = undefined;
  let a5: null = null;
  let a6: unknown = undefined;
  let a7: any = undefined;
  
  let b1: boolean = null;
  let b2: number = null;
  let b3: string = null
  let b4: undefined = null;
  let b5: null = null;
  let b6: unknown = null;
  let b7: any = null;
  ```







### symbol 和 unique symbol

`symbol` 主要用作对象、类、接口的计算属性名。无论是字符串字面量，还是数字字面量都能够表示一个固定的值，但是 `symbol` 类型不同与其他原始类型，它不存在字面量形式。  

`unique symbol` 类型的设计初衷是作为一种变通方法， 让一个 `Symbol` 值具有字面量的性质，即仅表示一个固定的值。

- 在接口中，只能将 `unique symbol` 类型的常量作为计算属性名，而不能使用 `symbol` 类型的常量作为计算属性名。否则接口失去了继承的意义。

  ```javascript
  const a: unique symbol = Symbol();
  const b: symbol = Symbol();
  
  interface Foo {
      [a]: string;
      [b]: string;
      // 接口中的计算属性名称必须引用必须引用类型为文本类型或 "unique symbol" 的表达式
  }
  ```

- `unique symbol` 类型没有改变 `Symbol` 值没有字面量表示形式的事实。为了能够将某个 Symbol 值视作表示固定值的字面量，TypeScript对“unique symbol”类型和Symbol的值的使用添加了限制。
  - TypeScript 选择将一个 Symbol 值与声明它的标识符绑定在一起，并 通过绑定了该 Symbol 值的标识符来表示“Symbol字面量”。这种设计的前提是，Symbol 值与标识符的 绑定关系是不可变的。
  
    ```javascript
    // 必须使用 const 声明 `unique symbol` 类型变量。
    const a: unique symbol = Symbol();
    
    // 必须使用 readonly 声明 `unique symbol` 类型的属性。
    interface Foo {
        [a]: string;
        readonly b: unique symbol;
    }
    ```
    
  - `unique symbol` 类型的值只允许使用“Symbol()”函数 或“Symbol.for()”方法的返回值进行初始化，因为只有这样才能够“确保”引用了唯一的Symbol值。
  
    ```javascript
    const a: unique symbol = Symbol();
    const b: unique symbol = Symbol('desc');
    
    // 不能将类型 `typeof a` 分配给类型 `typeof c`。
    const c: unique symbol = a;
    ```
  
    当使用相同参数调用“Symbol.for()”方法实际上返回的是相同的Symbol值，但是由于设计局限性，TS无法识别这种情况。
  
    ```javascript
    const a: unique symbol = Symbol.for("desc");
    const b: unique symbol = Symbol.for("desc");
    
    // 此条件将始终返回 "false"
    if (a === b) {
        console.log('do something...')
    }
    ```
  
  - 由于`unique symbol`类型是 `symbol` 类型的子类型，因此可以 将 `unique symbol` 类型的值赋值给 symbol 类型。
  
    ```javascript
    const a: unique symbol = Symbol();
    
    const b: symbol = a;
    ```
  
  - 如果程序中未使用类型注解来明确定义是symbol类型还是“unique symbol”类型，那么TypeScript会自动地推断类型。
  
    ```javascript
    // a和b均为'symbol'类型，因为没有使用const声明
    let a = Symbol();
    let b = Symbol.for('');
    
    // c和d均为'unique symbol'类型
    const c = Symbol();
    const d = Symbol.for('');
    
    // e和f均为'symbol'类型，没有使用Symbol()或Symbol.for()初始化
    const e = a;
    const f = a;
    ```

## 特殊类型

在理解下面几种特殊类型之前，建议先理解一下[控制流分析](./语言思考/#控制流分析)

### any

`any` 表示 “可以是任何类型”。 当一个变量被声明为 `any` 类型，编译器就会跳过对这个变量的类型检查。这和我们正常写 javascript  是一样的。

就像下面的示例，编译期不会对 `user` 变量做任何类型检查，到了运行期才报错。

```ts
function parse(json: string): any {
  return JSON.parse(json);
}

const user = parse('{ "age": "18" }');
user.age.toFixed(); // 💥 运行时炸了
```
- 特点：
    - 可以把任何类型赋给它，也可以把它赋给任何类型（除了 `never`）。
    - 可以访问该类型上的任何属性或方法。
- 后果:
    - 失去了所有类型检查和自动补全
- 使用场景： 
    - 从老旧的 JavaScript 项目迁移到 TS。
    - 第三方库类型极其混乱

::: warning
any 必须是“最后一步”, 过度使用 any 会让 TypeScript 变成“ AnyScript ”
:::

### unknown

`unknown` 是 `any` 的安全升级版。它也表示“可以是任何类型”，但它有一个底线：在你没确定它到底是什么之前，你不准对它做任何操作。

在示例中，我们必须先判断`value`的类型，才能进行操作。

```ts
let value: unknown = "Hello";

// value.toUpperCase(); // ❌ 报错：你还没证明它是字符串呢！

if (typeof value === "string") {
    console.log(value.toUpperCase()); // ✅ 通过类型收窄，现在安全了
}
```
- 使用场景：
  接收来自 API 的返回结果、用户输入或动态内容，这些你无法预先确定类型但又想保持类型安全的地方。

如果你需要一个“可以存放任何属性，但属性值类型未知”的对象，不要直接用`unknown`，而应该用:
``` ts 
const user: Record<string, any> = {}; 
user.age = 25; // ✅ 允许
```
或者，你还想保持安全：
```ts
const safeStorage: Record<string, unknown> = {};
safeStorage.item = "Hidden value"; // ✅ 允许添加
// console.log(safeStorage.item.toUpperCase()); // ❌ 报错，取出来用的时候依然受 unknown 保护
```
### void 和 never

`void` 和 `never` 这两种类型其实是最令人迷惑的。它们最常见、最合理的使用场景是用作函数的返回类型。 虽然在 JavaScript 中，如果你不明确指定函数的返回值，它默认会返回 `undefined`，但是在 TypeScript 中 `void`、 `never` 和 `undefined` 并不等价。 

在 TypeScript 的设计中， 我们可以把`boolean`、`number`、`string` 等看做是“数据类型”，而`void`和`never`可以看作“状态类型”。

它们不是为了表示值的类型，而是为了从类型层面去描述代码执行的状态，我们可以简单的理解为函数执行的状态。

::: warning
这里的“数据类型”和“状态类型”只是为了让我们更方便的理解`void`和`never`。在 TypeScript 源码中，对于这两种类型的处理的确不同于其他类型，但是在 TypeScript 官方文档并没有做区分。
:::

##### void

在设计中，`void` 有两个方面的含义：
- 这个函数能够被顺利执行完，并且只要你有一条路径能够让函数到达终点。
- 在函数声明中表达的含义是：强制函数不允许有返回值；在函数类型中表达的含义是：我并不关心函数的返回值是什么，即使你有返回值。

在下面这几个例子中，我们说明一下这两个含义。
- 我们先说一下在函数声明中使用`void`的情况。
```ts
// #1 函数被正常执行完成，没有任何返回值;
function log(Logging: string): void {
    console.log(Logging);
}

// #2 函数有一个路径可以执行完成，并且没有任何返回值 
function logOne(a: string): void {
    if (a !== '1') {
        throw new Error('a is not 1');
    }
    
    console.log(a);
}

// #3 函数有路径能执行完成，但是有个 number 类型的返回值
function getNotOne(a: string): void {
    if (a !== '1') {
        // ❌ 报错：
        return a;
    }

    console.log('a is 1');
}
```
- 在函数类型中的使用情况， 即使在函数类型中明确表示了返回void，但是可以被返回void，fn返回的数字被忽略了。
```ts
type Fn = () => void

const fn: Fn = () => 123;

// ❌ 报错, TS 不能将类型 `void` 分配给类型 `number`。
const x: number = fn();
```

那为什么函数声明和函数类型中，对于返回的void值处理情况不一样呢？
```ts
const names = ["Alice", "Bob"];
const log = (name: string) => console.log(name);

// forEach 期待的回调是 (value: string) => void
// console.log 正好返回 void，一切完美。
names.forEach(log);

// 但如果这样写：
const list: number[] = [];
// push 返回的是 number，如果 TS 严格限制必须返回 void，
// 下面这行极其常见的简写就会报错：
names.forEach(name => list.push(name.length));
```

##### never

`never`表示的含义是，函数绝对没有任何路径可以到达函数的终点

```ts
// #1 没有任何路径让函数执行完成
function fail(message: string): never {
  throw new Error(message);
}

// #2 存在路径让函数执行完成
function fun(a: number): never {
  if (a) throw new Error()
  return a;
}
```

### enum

`enum` (枚举)允许我们定义一组命名的常量，在 JS 中我们通常使用对象声明一组常量。 **它是 TS 极少数几个不仅存在于类型层面，还会真实生成 JS 代码**的特性之一。

下面的例子展示了枚举的基本使用。
```typescript
enum Direction {
    UP =  'UP',
    DOWN = 'DOWN',
    LEFT = "LEFT",
    RIGHT = "RIGHT"
};

function move(direction: Direction) {
    switch (direction) {
        case Direction.UP:
            console.log("Move up");
            break;
        case Direction.DOWN:
            console.log("Move down");
            break;
        case Direction.LEFT:
            console.log("Move left");
            break;
        case Direction.RIGHT:
            console.log("Move right");
            break;
    }
};

move(Direction.UP);
```
#### 枚举的三种形态

按照枚举成员的类型，枚举可以被分为数值枚举、字符串枚举、异构枚举。

- 数值枚举

  枚举成员是**全部都是**数值类型，这是最默认的形态。
  ```javascript
  enum Direction {
      Up, // 0
      Down, // 1
      Left, // 2
      Right, // 3
  }
  
  enum Direction {
      Up, // 0
      Down, // 1
      Left = 11, // 11
      Right, // 12
  }
  ```
  - 当未赋初始值时，枚举成员的值默认会基于上一个值 +1，并且第一个枚举成员的默认初始值为 0。
  - 可以为枚举成员指定起始值，并且后续成员会基于上一个值 +1。

- 字符串枚举
  
  枚举成员**全部都是**字符串常量。通常使用字符串枚举，因为在调试时，字符串比数字更有意义。
  ```typescript
  enum Direction {
      Up = "UP",
      Down = "DOWN",
      Left = "LEFT",
      Right = "RIGHT",
  
      U = Up,
      D = Down,
      L = Left,
      R = Right,
  }
  ```
  - 字符串枚举没有自动递增，每个成员都必须初始化。

- 异构枚举
  即混合数字和字符串类型的枚举成员。**极其不推荐使用**，因为这会增加代码的阅读心智负担。

  ```javascript
  enum Direction {
      Up = 0,
      Down = "DOWN",
      Left = "LEFT",
      Right = 3,
  }

  enum Direction {
      Up = 0,
      Down, // 可自动推导
      Left , // 可自动推导
      Right = 'RIGHT',
      Center = 'CENTER', // 不可自动推导，必须初始化
  }
  ```
  - 异构枚举的成员**最好初始化**。
  - 只有前面是数字，后面的成员才可以自动推导。也就是说只有“纯数字常量链”可以自动递增。
  - 一旦前面出现“字符串 or 计算值”，后面基本都要手动初始化。

#### 枚举本质和反向映射

前面我们说枚举会被编译成 JS 代码，让我们看一下这个过程是怎样的。

1. 创建一个异构枚举类型。
    ```typescript
    enum Direction {
        Up = 0,
        Down,
        Left = 3,
        Right,
        Center = 'CENTER',
    }
    ```

2. 将该枚举对象编译为 JS 。
    ```javascript 
    var Direction;

    (function (Direction) {
        Direction[Direction["Up"] = 0] = "Up";
        Direction[Direction["Down"] = 1] = "Down";
        Direction[Direction["Left"] = 3] = "Left";
        Direction[Direction["Right"] = 4] = "Right";
        Direction["Center"] = "CENTER";
    })(Direction || (Direction = {}));
    ```
    可以看到枚举类型的最终编译成 JS 就是一个对象，然后通过一个立即执行函数赋值。

3. 查看该函数的执行结果，打印该对象。
    ```javascript
    {
        '0': 'Up',
        '1': 'Down',
        '3': 'Left',
        '4': 'Right',
        Up: 0,
        Down: 1,
        Left: 3,
        Right: 4,
        Center: 'CENTER'
    }
    ```

从上面的编译过程可以看出：枚举成员的初始化策略正如前面所说，默认从0开始，如果前面是数字，则按照顺序 +1。

其实还可以看出一个东西：**反向映射**。这是数字枚举成员独有的特性：你不仅可以通过“名字”找到“值”，还可以通过“值”找到“名字”。

其底层原理如上面例子，TS 会把数字枚举编译成一个“双向字典”对象。而字符串枚举不支持反向映射。

例如：
```typescript
console.log(Direction.Down); // 打印“1”
console.log(Direction[3]);   // 打印“Left”
```
    
#### 枚举的两个世界

了解了枚举在实际编译后是一个对象，再结合前面“枚举的基本使用”的例子，你会不会产生一个疑问：`move(direction: Direction)` 接收的 `direction` 参数不应该是个对象类型吗？为什么在实际调用的过程中可以传 `Direction.UP` 参数？

你之所以会产生这种疑惑，是因为在 TS 中，`Direction` 这个名字同时拥有**两个身份**，但它们分别属于两个不同的平行世界。

- **值世界：它是一个对象**
 
  在编译后的 JS 里，Direction 确实是一个真实存在的对象。
  - 你可以点出它的属性，如 `Direction.UP`。

- **类型世界：它是一个联合类型**
  
  在 TS 的类型系统中，`Direction` 代表的是这个枚举所有成员值的并集。
  - 当你为变量添加类型注解时, 如 `direction:Direction`，它等价于：
    ```typescript
    Direction.Up | Direction.Down | Direction.Left
    ```

简单来讲，当枚举当做值来用的时候，它就是一个对象，但是如果你把它当做类型来用，它就是一个联合类型。







#### 枚举成员推导和分类

虽然说枚举成员的推导和分类在实际开发中基本不会被用到，但简单的了解还是必要的。并且在理解枚举成员推导之前，首先需要了解的是枚举成员的分类。

记得我们在前面说的，TS 会基于规则，给没有初始化的枚举成员自动赋值吗？这其实涉及到枚举的另一个特性：**枚举成员推导**。

##### 枚举成员分类

按照枚举成员的值的表示方式，枚举成员可以分为两类：常量成员和计算成员。

- 常量成员
  1. 值为字面量（没有定义初始值、数字字面量、字符串字面量和不包含替换值的模板字面量）。
  2. 值为对其他字面量枚举成员的引用。
  3. 值为常量表达式。

- 计算成员

  除了上面的情况外，剩下的都是计算成员。例如：
  ```
  enum FileAccess {
    // 常量成员
    None,
    Read = 1 << 1,
    Write = 1 << 2,
    ReadWrite = Read | Write,
    // 计算成员
    G = "123".length,
  }
  ```
##### 枚举成员推导

了解枚举成员的分类之后，说枚举成员推导规则就很简单了：

> **只有“常量数值表达式”才支持自动推导,一旦出现“不可推导成员”，后面所有成员都必须手动初始化。**。

它的设计本质就是
> **只有“编译期可计算的常量”才能参与自动推导**


::: warning
我说的是“枚举成员的推导”，而不是“值推导”。在 TS 的其他地方，类型和值通常是分离的（类型擦除后值还在），但是在枚举中，推导是双向发生的。
::: 


#### 联合枚举

联合枚举不是一种语法，而是一种特定的“形态”：当一个枚举的**所有成员**都是“字面量成员”时，这个枚举会被当成“联合类型”使用。

一旦枚举进化为联合枚举，它会带来两个翻天覆地的变化：

##### 成员变成了“类型”

在普通枚举里，成员只是值。但在联合枚举里，每一个成员都可以直接作为类型来使用。

```typescript
  enum ShapeKind {
    Circle,
    Square,
  }
   
  interface Circle {
    kind: ShapeKind.Circle;
    radius: number;
  }
   
  interface Square {
    kind: ShapeKind.Square;
    sideLength: number;
  }
   
  let c: Circle = {
    kind: ShapeKind.Square,
    // Type 'ShapeKind.Square' is not assignable to type 'ShapeKind.Circle'.
    radius: 100,
  };
```
##### 枚举本身变成了“成员类型的并集”
 
由于编译器知道枚举只有那几个固定的成员类型，它可以进行穷尽性检查，进行类型检查。此时，`ShapeKind` 类型在编译器眼里等同于 `ShapeKind.Circle | ShapeKind.Square`。

```typescript
function getArea(s: Circle | Square) {
  switch (s.kind) {
    case ShapeKind.Circle:
      // 在这里，s 被精准收窄为 Circle 类型
      return Math.PI * s.radius ** 2;
    case ShapeKind.Square:
      // 在这里，s 被精准收窄为 Square 类型
      return s.sideLength ** 2;
  }
}
```

如果不是联合枚举会怎样？

```typescript
enum Bad {
  A = 1,
  B = Math.random()
}
```

由于 `Bad.B` 在运行时才能知道具体的值，就没有办法穷尽检查，或者类型收窄了。

#### 常量枚举
 
常量枚举存在的唯一目的就是：极致的性能优化。

如前面所说，对于普通枚举类型，TS 会在编译后的 JS 里生成一个真实的对象（IIFE 立即执行函数形式）。而**常量枚举会在编译阶段被彻底抹除**，所有引用的地方都会被替换为字面量值，因此，**在常量枚举只能使用常量表达式**。

下面是一个常量枚举的示例：

```typescript
const enum Directions {
  Up,
  Down,
  Left,
  Right
}

let directions = [Directions.Up, Directions.Down];
```

编译为 JS 后：
```javascript
var directions = [0 /* Directions.Up */, 1 /* Directions.Down */];
```
 
 ###### 常量枚举的坑

虽然常量枚举优化了性能，但是实际上“坑更大”。由于工程兼容问题，跨文件引用的时候可能直接废掉了。
```typescript 
export const enum A { X = 1 }
```

#### 实践中的枚举使用

在实际项目开发中，“弃用枚举” 已经成了一个非常普遍的共识，甚至被写入了一些大厂的和非官方的开发规范中。

1. 运行时污染：违背了 TS 的“类型擦除”原则
    
    TS 的绝大多数特性都是“透明”的，编译后即消失。但枚举是个例外，它会生成一段厚重的 JS 代码

2. Tree-shaking 的恶梦

    现代构建工具（如 Webpack, Rollup）通过 Tree-shaking 剔除未使用的代码。由于枚举编译后是一个 IIFE（立即执行函数），很多构建工具的静态分析器会保守地认为这个函数“可能有副作用”，从而拒绝将其剔除。这意味着即使你只用了枚举中的一个成员，整个枚举对象都会被打包进生产环境。

3. 工程兼容问题

##### 替代方案

在最推荐的做法是利用**字面量类型**。它既保留了枚举的语义化优点，又完美规避了上述所有缺点。

```typescript
// 1. 定义一个普通对象并锁死
const Status = {
  Active: "ACTIVE",
  Inactive: "INACTIVE",
} as const;

// 2. 自动推导出联合类型: "ACTIVE" | "INACTIVE"
type Status = typeof Status[keyof typeof Status];

// 3. 使用
function update(s: Status) {}
update(Status.Active); // ✅ 语义化
update("ACTIVE");
```

### 函数

如果你了解 Java 等强类型语言的话，你就会知道为什么人们会说“函数在 JS 中是一等公民”，你也会知道为什么 TS 会这么设计函数的类型。

在 JS 中，函数既可以被直接调用，也可以被当做参数传递，也可以被当做返回值。TS 对于函数类型的设计，遵循了这种使用场景，所以一方面对函数声明增加了类型支持，以支持函数调用时的类型检查；另一方面添加了新的函数类型，以支持函数作为函数和返回值时做类型检查。

#### 函数声明

函数主要是由参数、返回值和函数体组成，因此在函数的类型设计中，重要的就是为参数和返回值添加类型注解。

当然，在 JS 的函数中，还有一个令人头痛的东西: this的指向。

##### 参数

JS 中函数的参数有几个特点：
- 所有参数都是可选的。在函数调用时可以不传递某些参数，未被传递值的参数默认值是 `undefined`。如果你需要让参数变成必填，就必须在函数内部手动实现参数的必填校验。
- 参数是有顺序的。不能跳过前面的参数只传后面的，所以你想跳过前面的参数只传后面的参数，就必须显式使用`undefined`
- 允许为参数添加默认值。
- 参数的数量是不固定的。ES6 中引入了剩余参数（ `...rest` ）语法接收任意数量的参数。

TS 不会改变 JS 的语言特性，但是从类型系统的角度，做了些必要的限制。
- 常规参数。可以添加类型注解，这也意味着 TS 的参数默认都是必选参数。
  ```typescript
  function add(x: number, y: number): number {
      return x + y;
  }
  ```
  但是如果没有为参数添加类型注解，编译器也无法静态推断其类型，参数默认为any类型。
  ```typescript
  // 参数“x”隐式具有“any”类型。
  function add(x, y) {
      return x + y;
  }
  ```
- 可选参数。
  可选参数应该位于参数列表的最后。
  ``` typescript 
  function add(x: number, y?: number): number {
    return y ? x + y : x;
  }
  ```
  严格模式下，TS 会自动为可选参数添加`undefined`类型，也就是说上面的可选参数`y`的类型是`number | undefined`。

- 默认参数。为参数添加默认值和 JS 没有什么区别。
  ``` typescript 
  function add(x: number = 3, y?: number): number {
    return y ? x + y : x;
  }
  ```
  但是默认参数有几个特点：
  - 具有默认值的参数，可以不显式定义类型，编译器可以自动推断其类型。
  - 具有默认值的参数，且位于参数列表末尾，会被视为可选类型；如果其不位于参数类别末尾，依然会被当做必选参数；可选参数不允许被赋予默认值。

- 剩余参数。可以为剩余参数添加类型注解，但是根据多个剩余参数类型是否一样，把剩余参数注解为数组类型或者元组类型。
  ```typescript
  // 把剩余参数注解为数组类型
  function add1(a: number, b: number, ...args: number[]): void {
      console.log(args);
  }

  // 把剩余参数注解为元组类型。
  function add2(a: number, b: number, ...args: [number, string]):void {
    console.log(args)
  }
  ```


##### this 类型

曾经 JS 最令人头痛的问题之一就是函数的 `this` 指向问题，ES6 引入了箭头函数来解决这个问题。TS 引入了 `this: type` 来指定函数的 `this` 的类型。这个能力主要是为“普通函数”准备的，箭头函数里的 `this` 类型通常由上下文自动推断。

``` typescript
type User =  { 
  name: string 
}

function getUsername(this: User) {
  return this.name;
}
```
#### 函数类型

JS 中函数不总是被直接调用的，很多时候函数是会被当做参数、返回值传递的。因此，TS 必须要添加一个**函数类型**，来支持函数作为参数、返回值时的类型检查。

在大多数静态语言（如 Java 或 C#）中，函数就是函数，对象就是对象，两者界限分明，但是在JS 的底层逻辑里，函数本质上就是一种可以被调用的对象。

TS 为了完美描述这种“亦函数亦对象”的特性，设计了两种不同的表达方式：普通调用签名和对象调用签名。

##### 普通调用签名

也可以被称为函数类型表达式，和前面的函数声明一样，函数类型也主要是为参数、返回值、`this` 添加类型注解。

```typescript
type fn = (a: number, b: number) => number;
```

##### 对象调用签名

在 JS 的生态里，有很多场景要求“本体是函数，同时也得存数据”。一个典型的例子如下。
```javascript
function counter(num) {
  counter.count += num;
}
counter.count = 0;
```

还有一个更典型的例子是`jQuery`库里的`$`。
- 你可以像函数一样调用它：`$('div')`。
- 它身上也挂满了各种工具方法：`$.ajax()`、`$.get()`。

如果没有对象调用签名，TS 就没法描述 `$` 这个既是函数又是命名空间的存在。对象调用签名可以使用**类型别名**或者**接口**进行描述。

```typescript
interface counter {
    (num: number): number,
    count: number,
};

const add: counter = function (num: number): number {
    return count + b;
};
```

#### 构造签名

JS 还有一个极其离谱的设计：函数既可以被直接调用，也可以被 `new` 实例化。(写到这里我真想吐槽下:shit:)。JS 中除了箭头函数外，几乎所有普通函数都可以被 `new`。虽然在实际开发中，我们通常会明确区分普通函数和构造函数。

TS 的设计相当严谨，充分考虑了函数被 `new` 实例化的场景。其设计构造签名的核心目的，就是为了在静态层面彻底区分“执行逻辑”与“生产对象”这两种完全不同的行为。如果一个函数在设计时是为了被 `new` 的，但调用者不小心直接执行了它（或者反之），在运行时往往会报错或产生 `undefined`。 

构造签名和[函数类型](#函数类型)说的两种签名方式是一样的。但是允许我们在类型层面加一把锁：只有具备 `new` 签名的类型，才能被实例化。下面是构造签名的两种声明方式。

- 普通构造签名
```typescript 
type fn = new (name: string) => string
```
- 对象构造签名
```typescript 
interface fn {
    new (a: number, b: number): number;
    (a: number, b: number): number;
}
```

#### 函数重载

在 JS 中，一个函数根据参数不同（个数不同、类型不同），可以有完全不同的行为和返回值，这是一种非常常见的“骚操作”。如果没有重载，TS 就无法精准描述这种输入与输出之间的对应关系。

这里有一个例子：假设你有一个 `double` 函数：输入数字返回数字，输入字符串返回字符串。如果不使用重载，你可能会这样写：
```typescript
function double(x: string | number): string | number {
    return typeof x === "string" ? x + x : x * 2;
}

const result = double(10); // result 的类型是 string | number
```
**痛点在这里**：编译器虽然知道 `result` 可能是数字或字符串，但它弄丢了因果关系。它不知道“因为我传了数字，所以结果一定是数字”。你之后调用 `result.toFixed()` 就会报错，因为 `string` 没这个方法。


函数重载的作用就是建立一套“声明-对应”的协议。它由两部分组成：
- 重载签名 (Overload Signatures)。这是给调用者看的“菜单”。它只声明各种合法的调用方式，没有具体的代码。
- 实现签名 (Implementation Signature)
这是给函数内部写的“工厂”。它必须足够宽泛，能兼容所有重载签名，但对外是不可见的。

```typescript
// 重载签名：精确描述因果
function double(x: number): number;
function double(x: string): string;

// 实现签名：负责逻辑分发
function double(x: any): any {
    return typeof x === "string" ? x + x : x * 2;
}

const n = double(10);     // TS 确信 n 是 number
const s = double("hi");   // TS 确信 s 是 string
```

::: warning
你是不是有个疑问：为什么 TS 没有像其他语言那样，每个重载函数都有各自的实现？那是因为在原生的 JS 中，是没有函数重载的概念的，它不允许存在重复的方法。TS 通过这种巧妙的方式实现了静态的类型检查。
:::
